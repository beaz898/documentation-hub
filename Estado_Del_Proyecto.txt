ESTADO DEL PROYECTO — Doclity
========================================
Última actualización: 19/05/2026


────────────────────────────────────────────────────────────────
RESUMEN EJECUTIVO
────────────────────────────────────────────────────────────────

Doclity es un SaaS multi-tenant para gestión y análisis inteligente
de documentación empresarial. Los usuarios suben documentos, hacen preguntas
en lenguaje natural y reciben respuestas basadas en el contenido. El sistema
detecta automáticamente contradicciones, duplicados y problemas de calidad
entre documentos del corpus.

Estado general: PRODUCCIÓN (desplegado en Vercel, conectado a Supabase + Pinecone)


────────────────────────────────────────────────────────────────
STACK TECNOLÓGICO
────────────────────────────────────────────────────────────────

Frontend    : Next.js 15 (React 19) + Tailwind CSS
LLM Rápido  : Claude Haiku 4.5 (chat, preguntas, reescritura de queries)
LLM Análisis: Claude Sonnet 4.6 (análisis de contradicciones, double-check)
Vector DB   : Pinecone serverless (multilingual-e5-large, 1024 dims)
Auth + DB   : Supabase (PostgreSQL + Row Level Security)
Facturación : Stripe (créditos prepago + planes de suscripción)
Deploy      : Vercel (auto-deploy desde GitHub)
Worker      : Node.js en Railway (análisis exhaustivo > 30s)


────────────────────────────────────────────────────────────────
BASE DE DATOS — SUPABASE
────────────────────────────────────────────────────────────────

Tablas principales (supabase-setup.sql)
  documents
    - id, name, size_bytes, chunk_count, org_id, user_id
    - status, source ('manual' | 'google_drive' | 'onedrive')
    - content_hash (SHA-256 para detectar duplicados exactos)
    - full_text (texto completo para RAG sin cortes de chunk)
    - created_at, updated_at
    - RLS: usuarios ven solo documentos de su org_id

  organizations
    - id, name, plan ('free'|'starter'|'pro'|'business'|'business_plus'|'enterprise')
    - credits, credits_extra (consumo atómico vía RPC consume_credits)
    - subscription_id, canceled_at, grace_period_ends_at
    - stripe_customer_id
    - upload_locked_by, upload_locked_at (semáforo de subidas simultáneas)

  memberships
    - user_id, org_id, role ('admin' | 'member')
    - Base del modelo multi-tenant

  invitations
    - id, org_id, email, token, status, expires_at
    - Para invitar miembros por email

  analysis_jobs
    - id, org_id, status (pending/processing/completed/failed)
    - document_name, input_data, result (JSONB), error_message
    - created_at, started_at, completed_at
    - Para análisis exhaustivos que superan timeout de Vercel

  feedback
    - user_id, org_id, message
    - Mensajes de feedback enviados por los usuarios

Tablas de analítica (supabase-analysis-persistence.sql)
  analysis_results
    - org_id, user_id, document_name, analysis_type (quick/exhaustive/style)
    - contradictions_found, contradictions_confirmed
    - minor_inconsistencies_found
    - duplicates_found, overlaps_found, style_problems_found
    - recommendation (INDEXAR / REVISAR / NO_INDEXAR)
    - involved_documents (JSONB: array de nombres de documentos involucrados)
    - RLS por org_id

  chat_queries
    - org_id, user_id, question, answer_length
    - documents_used (JSONB): [{documentId, documentName, score,
      chunks: number[], totalChunks: number}]
      Los campos chunks y totalChunks permiten calcular cobertura de corpus
    - RLS por org_id

PIPELINE DE ANÁLISIS
────────────────────────────────────────────────────────────────

Archivo principal: lib/analysis/pipeline.ts

Fases (modo quick)
  1. Hash check         — Detecta duplicados exactos por SHA-256
  2. Extract claims     — Haiku extrae afirmaciones atómicas del documento nuevo
  3. Pinecone retrieval — Busca fragmentos del corpus para cada afirmación
  3b. Rerank            — Haiku filtra candidatos irrelevantes (máx 6 seleccionados)
  4a. Judge             — Haiku compara afirmaciones vs corpus (contradicciones)
  4b. Verify claims     — Haiku verifica afirmaciones atómicas individuales
  5. Double-check       — Sonnet confirma/descarta candidatas (lote 1: 15, max 50)
  6. Synthesize         — Genera resultado final: discrepancies, minorInconsistencies,
                          overlaps, isDuplicate, styleProblems, recommendation

Fases adicionales (modo exhaustive)
  - Compara el documento nuevo contra todos los documentos del corpus
  - Procesado en worker (Railway) para evitar timeout de Vercel
  - Hasta 25 pares de documentos comparados (MAX_SELECTED_EXHAUSTIVE=25)
  - Rerank exhaustivo: máximo 10 candidatos seleccionados

Coste del análisis exhaustivo (30 cr base, con reembolso)
  Reanálisis (excludeFingerprints no vacío) con <2 contradicciones
    → reembolso 20 cr → coste final 10 cr (todos los planes)
  Planes Business/Business+/Enterprise — precio variable según resultado:
    · Ligero (<10 contradicciones)         → reembolso 10 cr → coste final 20
    · Medio (10-30 contradicciones)        → reembolso 5 cr → coste final 25
    · Pesado (>30 o cap 50 candidatas)     → sin reembolso → coste final 30
  Reanálisis con ≥2 contradicciones en Business+ aplica precio variable

Niveles de severidad
  'contradiction'         — Contradicción confirmada (confidence='alta')
                            Mostrada en sección principal de discrepancias
  'minor_inconsistency'   — Inconsistencia menor (confidence='posible')
                            Diferencia real pero ambas afirmaciones compatibles
                            Mostrada en sección separada con tono visual distinto

Taxonomía de errores detectados
  OMISIÓN SIGNIFICATIVA   — Lista incompleta respecto al corpus
  DISTORSIÓN CONCEPTUAL   — Términos técnicamente incorrectos
  SUSTITUCIÓN             — Reemplaza un término clave por otro
  EXAGERACIÓN             — Convierte matiz del corpus en afirmación absoluta
  DEGRADACIÓN             — Rebaja importancia de algo fundamental

Double-check progresivo (lib/analysis/double-check.ts)
  - Lote 1: 15 candidatas → si se alcanzan suficientes confirmadas, para
  - Lote 2: 10 candidatas adicionales si hace falta (modo rápido)
  - Modo exhaustivo: verifica TODAS, en lotes de 15 hasta completar
  - Máximo global: 50 candidatas (campo candidatesOverLimit en respuesta)
  - Memoria de descartes: excludeFingerprints evita re-verificar
    contradicciones ya descartadas en análisis anteriores

Costes de créditos
  /api/ask               : 1 crédito
  /api/analyze-v2 quick  : 5 créditos
  /api/analyze-v2 exhaust: 30 créditos (con posible reembolso parcial)
  /api/analyze-style     : 2 créditos
  /api/improve           : 1 crédito

────────────────────────────────────────────────────────────────
MOTOR RAG (lib/rag.ts)
────────────────────────────────────────────────────────────────

Flujo de consulta:
  1. Si hay historial de conversación, Haiku reescribe la pregunta en una
     query autónoma con palabras clave para Pinecone (rewriteQueryForSearch).
     La pregunta original se sigue usando para la respuesta final.
     Esta llamada es interna, no facturable.
  2. Embed query → Pinecone search (top 15, min score 0.3)
  3. Deduplica a ≤4 documentos fuente
  4. Obtiene texto completo desde Supabase (evita cortes de chunk)
  5. Llama a Haiku con documentos completos + historial (máx 6 mensajes)
  6. Devuelve answer + sources con chunks[] y totalChunks por documento

System prompt incluye reglas de honestidad: si el usuario pregunta si
la respuesta vino de los documentos, Claude debe responder con transparencia,
distinguiendo entre información del corpus y posible reformulación propia.

────────────────────────────────────────────────────────────────
GOTCHAS
────────────────────────────────────────────────────────────────

- Documentos antiguos pueden no tener full_text en Supabase → fallback a chunks.
- Pinecone free tier: rate-limit 250K tokens/min; lib/embeddings.ts auto-espera 60s.
- RLS usa service role en servidor; nunca usar el cliente anon para escrituras.
- chat_queries anteriores a commit b71dd5e (mayo 2026) no tienen chunks/totalChunks.
- ImprovementModal: handleSolveOne/handleSolveGroup resuelven duplicidades localmente
  (sin API, sin crédito) usando findParagraphContaining().
- El worker en Railway necesita las variables de entorno SUPABASE_URL,
  SUPABASE_SERVICE_ROLE_KEY, PINECONE_API_KEY y ANTHROPIC_API_KEY.
- El coste del análisis exhaustivo es 30 cr; el reembolso se aplica al terminar,
  no al inicio. Si el worker falla, los créditos no se reembolsan automáticamente.
- getOrgFeatures() debe consultarse antes de mostrar funcionalidades de Drive
  o el panel de analytics: un plan free no tiene acceso a esas páginas.
