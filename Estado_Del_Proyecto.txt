ESTADO DEL PROYECTO — Documentation Hub
========================================
Última actualización: 11/05/2026


────────────────────────────────────────────────────────────────
RESUMEN EJECUTIVO
────────────────────────────────────────────────────────────────

Documentation Hub es un SaaS multi-tenant para gestión y análisis inteligente
de documentación empresarial. Los usuarios suben documentos, hacen preguntas
en lenguaje natural y reciben respuestas basadas en el contenido. El sistema
detecta automáticamente contradicciones, duplicados y problemas de calidad
entre documentos del corpus.

Estado general: PRODUCCIÓN (desplegado en Vercel, conectado a Supabase + Pinecone)


────────────────────────────────────────────────────────────────
STACK TECNOLÓGICO
────────────────────────────────────────────────────────────────

Frontend    : Next.js 15 (React 19) + Tailwind CSS
LLM Rápido  : Claude Haiku 4.5 (chat, preguntas)
LLM Análisis: Claude Sonnet 4.6 (análisis de contradicciones, double-check)
Vector DB   : Pinecone serverless (multilingual-e5-large, 1024 dims)
Auth + DB   : Supabase (PostgreSQL + Row Level Security)
Facturación : Stripe (créditos prepago + planes de suscripción)
Deploy      : Vercel (auto-deploy desde GitHub)
Worker      : Node.js separado (análisis exhaustivo > 30s)


────────────────────────────────────────────────────────────────
BASE DE DATOS — SUPABASE
────────────────────────────────────────────────────────────────

Tablas principales (supabase-setup.sql)
  documents
    - id, name, size_bytes, chunk_count, org_id, user_id
    - status, source ('manual' | 'google_drive')
    - content_hash (SHA-256 para detectar duplicados exactos)
    - full_text (texto completo para RAG sin cortes de chunk)
    - created_at, updated_at
    - RLS: usuarios ven solo documentos de su org_id

  organizations
    - id, name, subscription_id, canceled_at, grace_period_ends_at
    - credits, extra_credits (consumo atómico vía RPC consume_credits)

  memberships
    - user_id, org_id, role ('admin' | 'member')
    - Base del modelo multi-tenant

  analysis_jobs
    - id, org_id, status (pending/processing/completed/failed)
    - input_data, result_data (JSONB)
    - Para análisis exhaustivos que superan timeout de Vercel

  upload_lock
    - org_id, locked_by_user_id, locked_at
    - Semáforo: impide subidas simultáneas en la misma org

Tablas de analítica (supabase-analysis-persistence.sql)
  analysis_results
    - org_id, user_id, document_name, analysis_type (quick/exhaustive/style)
    - contradictions_found, contradictions_confirmed
    - minor_inconsistencies_found (novedad 11/05/2026)
    - duplicates_found, overlaps_found, style_problems_found
    - recommendation (INDEXAR / REVISAR / NO_INDEXAR)
    - involved_documents (JSONB: array de nombres de documentos involucrados)
    - RLS por org_id

  chat_queries
    - org_id, user_id, question, answer_length
    - documents_used (JSONB): [{documentId, documentName, score,
      chunks: number[], totalChunks: number}]
      Los campos chunks y totalChunks permiten calcular cobertura de corpus
    - RLS por org_id

PIPELINE DE ANÁLISIS
────────────────────────────────────────────────────────────────

Archivo principal: lib/analysis/pipeline.ts

Fases (modo quick)
  1. Hash check         — Detecta duplicados exactos por SHA-256
  2. Extract claims     — Haiku extrae afirmaciones atómicas del documento nuevo
  3. Pinecone retrieval — Busca fragmentos del corpus para cada afirmación
  4a. Judge             — Haiku compara afirmaciones vs corpus (contradicciones)
  4b. Verify claims     — Haiku verifica afirmaciones atómicas individuales
  5. Double-check       — Sonnet confirma/descarta candidatas (lote 1: 15, max 50)
  6. Synthesize         — Genera resultado final: discrepancies, minorInconsistencies,
                          overlaps, isDuplicate, styleProblems, recommendation

Fases adicionales (modo exhaustive)
  - Compara el documento nuevo contra todos los documentos del corpus
  - Procesado en worker separado para evitar timeout de Vercel
  - Hasta 25 pares de documentos comparados (MAX_SELECTED_EXHAUSTIVE=25)

Niveles de severidad (novedad)
  'contradiction'         — Contradicción confirmada (confidence='alta')
                            Mostrada en sección principal de discrepancias
  'minor_inconsistency'   — Inconsistencia menor (confidence='posible')
                            Diferencia real pero ambas afirmaciones compatibles
                            Mostrada en sección separada con tono visual distinto

Taxonomía de errores detectados (añadida 11/05/2026)
  OMISIÓN SIGNIFICATIVA   — Lista incompleta respecto al corpus
  DISTORSIÓN CONCEPTUAL   — Términos técnicamente incorrectos
  SUSTITUCIÓN             — Reemplaza un término clave por otro
  EXAGERACIÓN             — Convierte matiz del corpus en afirmación absoluta
  DEGRADACIÓN             — Rebaja importancia de algo fundamental

Double-check progresivo (lib/analysis/double-check.ts)
  - Lote 1: 15 candidatas → si se alcanzan suficientes confirmadas, para
  - Lote 2: 10 candidatas adicionales si hace falta (modo rápido)
  - Modo exhaustivo: verifica TODAS, en lotes de 15 hasta completar
  - Máximo global: 50 candidatas (campo candidatesOverLimit en respuesta)
  - Memoria de descartes: excludeFingerprints evita re-verificar
    contradicciones ya descartadas en análisis anteriores

Costes de créditos
  /api/ask               : 1 crédito
  /api/analyze-v2 quick  : 5 créditos
  /api/analyze-v2 exhaust: 20 créditos
  /api/analyze-style     : 2 créditos
  /api/improve           : 1 crédito
