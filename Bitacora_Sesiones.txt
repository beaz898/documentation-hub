────────────────────────────────────────────────────────────────
SESIÓN 1 — 03/04/2026
Título: Proyecto inicial — RAG funcional
────────────────────────────────────────────────────────────────

OBJETIVO
Construir la base del producto: subir documentos, indexarlos y poder hacer
preguntas en lenguaje natural que se respondan usando el contenido.

CAMBIOS IMPLEMENTADOS

Motor RAG (lib/rag.ts)
  - Embedding de la pregunta → búsqueda en Pinecone (topK=15, score mínimo 0.3)
  - Deduplicación a ≤4 documentos más relevantes
  - Obtención de texto completo desde Supabase (evita cortes de chunk)
  - Llamada a Claude con contexto completo + historial de conversación
  - Soporte de historial (últimos 6 mensajes)

Ingesta de documentos (app/api/ingest/route.ts)
  - Extracción de texto: PDF (pdf-parse), DOCX (mammoth), TXT/MD/CSV/JSON/HTML
  - Chunking: fragmentos de ~2000 chars con overlap de 200 chars
  - Embeddings con Pinecone Inference API (multilingual-e5-large, 1024 dims)
  - Upsert en Pinecone con metadata: documentId, documentName, chunkIndex,
    totalChunks, orgId
  - Registro en Supabase: tabla documents con id, name, size_bytes, chunk_count,
    org_id, user_id, status

Interfaz de chat (app/chat/page.tsx)
  - Mensaje con soporte Markdown (componente ChatMessage.tsx)
  - Sidebar de documentos (DocumentsSidebar.tsx): subir, listar, eliminar

Base de datos (supabase-setup.sql)
  - Tabla documents con RLS por org_id
  - Índices en org_id, user_id, created_at

STACK DEFINIDO
  - Frontend: Next.js 15 + Tailwind CSS → Vercel
  - LLM: Claude (Anthropic API)
  - Vector DB: Pinecone serverless
  - Auth + DB: Supabase

7. MEJORA DE PROMPTS: TAXONOMÍA DE ERRORES

   Los tres LLMs del pipeline recibieron instrucciones ampliadas para detectar:

   OMISIÓN SIGNIFICATIVA
   - El documento nuevo presenta una lista incompleta respecto al corpus
   - Ejemplo: "hay 2 tipos" cuando el corpus define 3 o más
   - Veredicto: contradicción

   DISTORSIÓN CONCEPTUAL / SUSTITUCIÓN
   - Usa términos técnicamente diferentes para el mismo concepto
   - Ejemplo: "cognitiva" cuando el corpus dice "inteligente"
   - Veredicto: inconsistencia menor si el significado es cercano,
     contradicción si cambia el sentido

   EXAGERACIÓN
   - Convierte un matiz del corpus en afirmación absoluta
   - Ejemplo: "elimina todos los errores" cuando el corpus dice "reduce errores"
   - Veredicto: contradicción

   DEGRADACIÓN
   - Presenta como secundario algo que el corpus define como fundamental
   - Veredicto: contradicción

   Archivos modificados: lib/analysis/judge.ts, verify-claims.ts, double-check.ts

8. RESOLUCIÓN DIRECTA DE DUPLICIDADES (SIN LLM)

   Antes
   - Click en "Solventar" duplicidad → llamada a /api/improve → gasta 1 crédito
   - El LLM generaba la propuesta de borrado

   Ahora
   - Click en "Solventar" → propuesta generada localmente en el frontend
   - Se detecta el párrafo completo que contiene el fragmento duplicado
   - Se propone {find: párrafo_completo, replace: ''} directamente
   - Sin llamada API, sin consumo de créditos
   - Detección de párrafo: busca \n\n antes y después del fragmento duplicado

   Archivos: components/ImprovementModal.tsx (handleSolveOne, handleSolveGroup)

9. AGRUPACIÓN DE DUPLICIDADES POR DOCUMENTO

   - Cada sub-grupo tiene su propio botón "Resolver (N fragmentos)"
   - Estado de colapso independiente por sub-grupo (collapsedSubGroups: Set<string>)

OBJETIVO
Corregir bugs descubiertos en pruebas de producción del nuevo modelo de

CAMBIOS IMPLEMENTADOS

1. FIX: RACE CONDITIONS DEL RUNNER (4 fixes simultáneos)

   a) Claim exclusivo en conv-handler.ts (FIX 1)
      - Antes: SELECT en 'running' + UPDATE posterior → ventana de race
      - Ahora: UPDATE condicional WHERE locked_at IS NULL OR locked_at < threshold
        → solo el primer runner obtiene filas; el segundo obtiene claimed=[]

   b) Append atómico de steps en persist-conv.ts (FIX 2)
      - Antes: SELECT steps → push → UPDATE (read-modify-write; race con otros runners)
      - Ahora: RPC append_step_to_message → UPDATE steps = steps || jsonb_build_array()
        → atómico a nivel de base de datos; lanza en error (no silencia)

   c) Heartbeat + orphan filtering en runner-conv.ts (FIX 3)
      - refreshHeartbeat() actualiza locked_at al inicio de cada iteración ReAct
      - buildStepsIntoMessages usa pairedIds para excluir tool_calls sin result
        (síntoma de race condition anterior)
      - validateHistory: mensajes 'failed' solo loguean (no abortan el turno)

   d) SendBtn onClick en ConversationInput.tsx (FIX 4)
      - El botón de enviar no tenía prop onClick → solo Enter funcionaba
      - Añadido onClick a SendBtn y pasado en los tres puntos de uso

2. FIX: TYPO EN adjustCredits (lib/credits.ts)
   - adjustCredits usaba .select('credits') y .update({ credits: newBalance })
   - La columna correcta es credits_remaining → silenciaba el error sin actualizar
   - Corrección: .select('credits_remaining') y .update({ credits_remaining: newBalance })

3. FIX: STATUS INCORRECTO PARA ESCALATION (lib/agent/runner-conv.ts)
   - handlePauseResult asignaba awaiting_user a type='escalation'
   - ConversationInput maneja escalation bajo awaiting_confirmation
   - Fix: ternario isConfirmationLike = type === 'confirmation' || type === 'escalation'
     → escalation ahora recibe status awaiting_confirmation correctamente

4. REDUCCIÓN DE LATENCIA: POLL_INTERVAL 5s → 1s (worker/src/index.ts)
   - Bajado de 5000ms a 1000ms
   - Aplica a los tres pollers: analysis_jobs, conv turns, agent_tasks
   - Latencia máxima de reanudación: 5s → 1s (media ~500ms)
   - Carga extra: ~96 queries/min adicionales en idle (asumible en Supabase Pro)
   - El HTTP trigger directo Vercel→Railway (Opción D) se planifica para después

5. FIX CRÍTICO: locked_at NO SE RESETEABA AL REANUDAR
   Causa raíz
   - Cuando el runner pausa (awaiting_confirmation), el heartbeat deja locked_at
     con un valor reciente (p.ej. now()-30s).
   - Al reanudar, el endpoint llamaba updateMessageStatus('running') sin tocar locked_at.
   - El worker evaluaba locked_at IS NULL → false; locked_at < threshold → false
     → No reclama el mensaje. Espera hasta que locked_at envejece 5 min → STUCK threshold.
   - Resultado observado: ~4m37s de espera entre que el usuario aprobaba y el worker procesaba.
   Solución
   - message/route.ts: updateMessageStatus(…, 'running', { locked_at: null })
   - locked_at=NULL → el worker lo reclama en ≤1s por la vía rápida

6. FIX: MODIFY DEJABA tool_use SIN tool_result → HTTP 400 ANTHROPIC
   Causa raíz
   - Al responder "Modificar", el endpoint añadía: confirmation_response(modify) + user_message
   - El pre-bucle de runner-conv veía el tool_call pendiente y lo ejecutaba con el input
     ORIGINAL (igual que approve), añadiendo tool_result DESPUÉS del user_message en steps[]
   - buildStepsIntoMessages escanea tool_results parando al primer non-meta, non-tool_result step
     → user_message cortaba el escaneo ANTES de llegar al tool_result
   - El tool_call estaba en pairedIds (result existía) → incluido en bloque assistant
   - Pero el tool_result no era recogido → Anthropic recibía tool_use sin result → HTTP 400
   Solución (runner-conv.ts, pre-bucle)
   - Para cada pending tool_call, buscar confirmation_response(modify) en steps.slice(tcIdx)
   - Si wasModified → skip: no ejecutar la tool original, no añadir tool_result
   - Resultado: tool_call ∉ pairedIds → excluido del historial
   - El LLM recibe: [think text] + [user_message de modificación] → replantea desde cero
   - Detección por slice(tcIdx): robusta ante múltiples confirmaciones en el mismo turno
