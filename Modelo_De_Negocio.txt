MODELO DE NEGOCIO — Doclity
======================================
Última actualización: 27/05/2026


────────────────────────────────────────────────────────────────
PROPUESTA DE VALOR
────────────────────────────────────────────────────────────────

Doclity ayuda a empresas a mantener su documentación interna
coherente y consultable. El sistema:

1. Responde preguntas sobre la documentación en segundos (RAG)
2. Detecta contradicciones, duplicados e inconsistencias automáticamente
3. Propone mejoras concretas que el usuario puede aplicar en un clic
4. Muestra estadísticas de uso: qué se consulta, qué no, qué porcentaje
   del corpus está siendo realmente aprovechado

Público objetivo: equipos de conocimiento, áreas de calidad, RRHH, legal,
y cualquier equipo que gestione documentación normativa o de procesos.

────────────────────────────────────────────────────────────────
MODELO DE CRÉDITOS
────────────────────────────────────────────────────────────────

El sistema usa créditos prepago por operación. Cada organización tiene
un saldo de créditos que se consume con cada acción de IA.

Costes por operación (CREDIT_COSTS en lib/credits.ts)
  Consulta de chat (/api/ask)              : 1 crédito
  Análisis de calidad rápido               : 5 créditos
  Análisis de calidad exhaustivo           : 30 créditos (base)
  Análisis de estilo (/api/analyze-style)  : 2 créditos
  Mejora de documento (/api/improve)       : 1 crédito
  Subida de documento (indexación)         : GRATIS
  Resolución de duplicidades               : GRATIS (propuesta local en frontend)

  Agente IA (/api/agent/run)               : 15–50 créditos (estimación al crear)
    - Base: 15 créditos
    - Objetivo >300 chars: +5 créditos
    - Modo step_by_step: +5 créditos adicionales
    - Máximo: 50 créditos
    - Al completar/cancelar: reconciliación con adjustCredits()
      (devuelve diferencia entre estimado y real basado en tokens)

Nota: el crédito se descuenta ANTES de ejecutar la operación para evitar
sobrepasar el límite con solicitudes paralelas (RPC atómica en Supabase).
Los reembolsos del análisis exhaustivo se aplican al finalizar el worker.


────────────────────────────────────────────────────────────────
PRECIO VARIABLE DEL ANÁLISIS EXHAUSTIVO
────────────────────────────────────────────────────────────────

El análisis exhaustivo siempre descuenta 30 créditos al inicio.
Al finalizar, el worker puede devolver créditos según dos reglas:

1. DESCUENTO DE REANÁLISIS (todos los planes)
   Condición: excludeFingerprints no vacío Y < 2 contradicciones confirmadas
   → Reembolso de 20 créditos → coste final: 10 créditos

2. PRECIO VARIABLE (solo Business, Business+, Enterprise)
   Aplica cuando el reanálisis tiene ≥2 contradicciones, o en análisis nuevos:
   · Ligero (<10 contradicciones)         → reembolso 10 → coste final: 20 cr
   · Medio (10-30 contradicciones)        → reembolso  5 → coste final: 25 cr
   · Pesado (>30 o cap de 50 candidatas)  → sin reembolso → coste final: 30 cr

   En planes sin precio variable (free, starter, pro), el coste siempre es 30 cr
   (salvo el descuento de reanálisis, que aplica a todos).

Implementación: worker/src/index.ts llama a refundCredits() al terminar.
Si el worker falla, los créditos no se reembolsan automáticamente.


────────────────────────────────────────────────────────────────
CRÉDITOS EXTRA
────────────────────────────────────────────────────────────────

Los usuarios pueden comprar créditos adicionales independientemente del plan:

  Pack disponible
    pack_500: 500 créditos — pago único vía Stripe Checkout

  Características
    - Solo accesible para el rol admin de la organización
    - Se acumulan en organizations.credits_extra (separados de credits del plan)
    - No caducan
    - El sistema consume primero credits del plan, luego credits_extra
      (lógica en la RPC consume_credits de Supabase)

  Endpoint: POST /api/billing/buy-credits


────────────────────────────────────────────────────────────────
FLUJO DE FACTURACIÓN IMPLEMENTADO
────────────────────────────────────────────────────────────────

Suscripción mensual
  1. Usuario accede a /settings/billing
  2. Elige plan → checkout Stripe → webhook actualiza organizations en Supabase
  3. Webhook de Stripe gestiona:
     - checkout.session.completed → activar plan, asignar créditos, asignar features
     - customer.subscription.updated → cambio de plan
     - customer.subscription.deleted → marcar canceled_at, iniciar grace period
     - invoice.payment_failed → notificar al admin (pendiente de UI)
     - checkout.session.completed (type=credit_pack) → sumar créditos extra

Compra de créditos extra
  - Pago único vía Stripe Checkout (mode: 'payment')
  - Webhook identifica type=credit_pack en metadata y suma a credits_extra

Grace period
  - 90 días tras cancelación antes de purgar datos
  - Tabla organizations: canceled_at + grace_period_ends_at
  - Durante grace period el acceso sigue activo
  - Al expirar: datos de Supabase + Pinecone eliminados, usuario de Auth eliminado
  - Purga automática ejecutada por /api/admin/purge-expired (cron manual o Vercel cron)


────────────────────────────────────────────────────────────────
GATES DE FUNCIONALIDADES (lib/plan-features.ts)
────────────────────────────────────────────────────────────────

La función getOrgFeatures(supabase, orgId) devuelve:
  {
    hasDrive: boolean           — Drive disponible en el plan
    hasAnalyticsPanel: boolean  — Panel /settings/usage disponible
    hasVariablePricing: boolean — Reembolso parcial en exhaustivo
    hasAgent: boolean           — Agente IA disponible (business, business+, enterprise)
    maxUsers: number | null     — Límite de miembros (null = ilimitado; enterprise)
    plan: string                — Nombre del plan actual
  }

  Nota: business_plus tiene maxUsers=25 (no ilimitado). Solo enterprise es null.

Uso: consultada antes de mostrar Drive, panel de analytics,
o calcular reembolsos. Si el plan no existe en PLAN_FEATURES, aplica
el fallback más restrictivo (free: hasDrive=false, maxUsers=1).

COSTES DE INFRAESTRUCTURA (ESTIMADOS)
────────────────────────────────────────────────────────────────

Pinecone serverless
  - Gratis: hasta 2GB storage, 100K vectores
  - Un documento de ~10 páginas genera ~10 vectores (1024 dims)
  - 100K vectores ≈ 10.000 documentos de tamaño medio

Supabase
  - Gratis: 500MB DB, 50K usuarios auth
  - Pro ($25/mes): 8GB DB, backups automáticos

Vercel
  - Gratis: 100GB bandwidth/mes, funciones serverless con timeout 30s
  - Pro ($20/mes): timeout hasta 300s, dominio personalizado

Railway (worker)
  - Proceso Node.js para análisis exhaustivos > 30s
  - Coste variable según uso de CPU/memoria

Anthropic (Claude)
  - Claude Haiku 4.5: ~$0.25 / M tokens input, ~$1.25 / M tokens output
  - Claude Sonnet 4.6: ~$3 / M tokens input, ~$15 / M tokens output
  - Un análisis rápido típico: ~15K tokens → ~$0.05 con Haiku, ~$0.10 con Sonnet
  - Un análisis exhaustivo: ~50K tokens → ~$0.15 Haiku + $0.50 Sonnet

Estimación para 100 usuarios activos / mes
  - 500 consultas de chat: ~$0.25 (Haiku)
  - 100 análisis rápidos: ~$10
  - 20 análisis exhaustivos: ~$14
  - Total API: ~$25/mes
  - Infraestructura (Vercel Pro + Supabase Pro + Railway): ~$60/mes
  - Total estimado: ~$85/mes para 100 usuarios activos
